### 2026-06-28: Modular Ecosystem Vision
- **Decision**: Adopt a long-term vision of a personal ecosystem of targeted devices, with the phone as the connectivity hub.
  - *Rationale*: Building one perfect do-everything phone is unrealistic as a first project. Decomposing into targeted modules (car system, etc.) makes each piece achievable. The phone remains the primary project.
- **Decision**: First envisioned module is a car system (navigation + music), connecting to phone via USB.
  - *Rationale*: Car dock scenario makes USB ideal — provides data + power in one cable, higher bandwidth than Bluetooth, lower latency. Phone charges while docked. Android USB tethering is native. Car system offloads the hardest work (cellular) to the phone.
- **Decision**: USB is the primary ecosystem interconnect; Bluetooth/WiFi deferred for future wireless modules.
  - *Rationale*: USB is simpler, higher bandwidth, and provides power. Wireless is needed only for modules that can't be physically docked — that's a future problem.
- **Decision**: Ecosystem vision does not change MVP scope, but does constrain hardware selection — MCU must have USB capability.
  - *Rationale*: Selecting an MCU without USB would preclude future ecosystem integration. This is a low-cost constraint (many candidate MCUs have USB) that keeps the door open.

### 2026-06-28: RTOS Selection
- **Decision**: Use Zephyr RTOS (not FreeRTOS).
  - *Rationale*: Every subsystem the phone needs is built into Zephyr: USB device stack (CDC ACM, ~~CDC ECM for tethering~~ [superseded — see USB HS/ULPI Revisit: tethering now uses the SIM7600's own USB port, not MCU CDC ECM], MSC for storage access), PPP networking over cellular modem, FatFs/LittleFS for SD card, LVGL for display, power management subsystem for battery policy, BLE stack. With FreeRTOS these would be 5+ separate third-party integrations (TinyUSB, lwIP, FatFs, LVGL, vendor BT stack) that don't always compose cleanly.
  - *Learning alignment*: User explicitly wants a challenging, time-consuming project that pushes beyond prior experience. Zephyr's devicetree + Kconfig + west + subsystems model teaches modern embedded industry practices (Linux-grade workflows) that transfer to Nordic, NXP, Intel, TI ecosystems. The steep learning curve is a feature, not a bug, for this project's goals.
  - *Ecosystem advantage*: ~~CDC ECM (Ethernet-over-USB) is a built-in Zephyr USB function — the phone-as-USB-Ethernet-device for the car module tethering scenario is a config flag, not a from-scratch USB class implementation.~~ **SUPERSEDED 2026-06-28**: The phone-as-USB-Ethernet-device (MCU bridging LTE to CDC ECM) is no longer the tethering architecture — the SIM7600's own USB port does RNDIS/ECM directly. The Zephyr CDC ECM function is no longer needed for tethering (though it remains available if a future use case wants MCU-originated Ethernet-over-USB). See "USB HS / ULPI Revisit" decision below.
  - *Nordic native path*: nRF Connect SDK is Zephyr-based. If nRF52840 is selected for MCU, Zephyr is the native RTOS.
  - *Tradeoff*: Zephyr has a steep learning curve (devicetree, Kconfig, west build system). Expect 1-2 weeks of ramp-up before productive phone firmware work. Mitigation: complete the "Practical Zephyr" tutorial series (memfault) and Shawn Hymel's Zephyr guide before starting phone firmware.
  - *Cellular modem note*: Neither Zephyr nor FreeRTOS has ready-made SIM7600 support. Zephyr's modem subsystem (modem_chat + CMUX + PPP + pipelink) provides a real architecture to plug a SIM7600 port into; FreeRTOS Cellular Library would require building that infrastructure from scratch. This is a wash on effort but Zephyr provides better scaffolding.
- **Decision**: Adopt a "no feature blocking" principle for component selection — all features rated 5+ on the wishlist must remain achievable (hardware-capable, even if not implemented in MVP).
  - *Rationale*: Avoiding rework and respins. If a component choice blocks a 5+ feature, we'd need to swap the component later (MCU change = PCB respin + firmware rewrite). Selecting components with headroom for 5+ features costs little upfront and preserves all options.
  - *5+ features that constrain MCU selection*: USB device (8), SPI/SDIO for MicroSD (7), BLE (6), I2S/audio DAC (6), camera interface (6), color display interface (5-6), video recording processing headroom (5).

### 2026-06-28: Modem Revisit (SIM7600 Selection Review)
- **Decision**: **Stick with SIMCom SIM7600 series** — the four concerns raised in the modem revisit prompt do not collectively justify switching. Two changes adopted.
  - *Rationale*: Four concerns were researched against SIM7600, EG25-G, EG912U-GL, and LARA-R6401:
    1. **Standby power**: The docs' "3 mA sleep" figure was deep-sleep (cannot receive calls), not idle/DRX. Actual SIM7600 LTE idle/DRX current is **17.5 mA** (datasheet: SIM7600E Hardware Design V1.00 Table 26). 24h standby needs ~420 mAh — achievable with an 800+ mAh battery. The modem still dominates standby; MCU sleep (~20 µA) is negligible. EG25-G is worse (22 mA); EG912U-GL is better (14 mA) but eliminated for the two-audio-path problem; LARA-R6401 claims ~1.1 mA (unverified). This concern does not justify switching.
    2. **B71 validation gap (RESOLVED)**: Waveshare makes a **SIM7600NA-H 4G HAT** (~$89) with B71 and the same NAU8810 codec. Switching the prototyping HAT from G-H to NA-H closes the gap — B71 can be validated early without a custom PCB. B71 is supplementary in the user's area (good coverage) but cheap to validate.
    3. **Simultaneous VoLTE + data (real risk, future feature)**: Forum evidence (TinyGSM #649, LilyGO #90) shows SIM7600 PDP context conflicts (IMS cid=2 interferes with data cid=1). No clear hobbyist evidence of successful simultaneous VoLTE+data on SIM7600. EG25-G has better-documented multi-PDP support (PinePhone-proven); LARA-R6401 has best multi-PDN docs. **However**: this is a future ecosystem feature (tethering while on a call), NOT an MVP or daily-driver requirement. The MVP (make/receive calls) is unaffected. Make this the first HAT test; if it fails, it blocks a future feature, not the core phone. Fallback: EG25-G (loses B71) or LARA-R6401 (supply risk).
    4. **LGA assembly (wash)**: All candidate LTE modules are LGA — none are hand-solderable. No LTE/VoLTE module exists in a hand-friendly package. NFR-3 ("hand-solderable for prototypes") is unachievable for any LTE module. Realistic path: JLCPCB assembly for the modem section (~$57–72), hand-solder the rest.
- **Change 1**: Switch prototyping HAT from SIM7600G-H to **SIM7600NA-H** (~$89) — resolves B71 validation gap. Has B71 + NAU8810 codec + GNSS.
- **Change 2**: Make **simultaneous VoLTE + data the first HAT test** (AT command sequence in research-notes.md modem revisit findings). If it fails after genuine effort, revisit with EG25-G or LARA-R6401 as fallbacks — but only for the ecosystem tethering feature, not the MVP.
- **Carrier note**: T-Mobile/Mint is a recommendation, not a lock-in. T-Mobile is most lenient with non-certified devices on prepaid. AT&T/Verizon are stricter on IMEI whitelisting. SIM7600A-H supports bands for all three major US carriers — switching later is possible.

The EG25-G would be safer for the ecosystem feature but loses B71 (the original reason for choosing SIM7600). The decision is to validate on the HAT first rather than switch speculatively.

### 2026-06-28: Modem Revisit (Second Round — Full Re-evaluation)
- **Decision**: **SIM7600 confirmed and locked.** No alternative module justifies switching. The modem selection is resolved.
  - *Rationale*: A deeper re-evaluation examined LARA-R6401 supply chain, EC25-AF as a new candidate, the EG25-G T-Mobile VoLTE issue, and the true scope of the SIM7600 PDP context conflict.
    1. **LARA-R6401 — DISQUALIFIED (911/E911 not supported)**: The LARA-R6401 datasheet explicitly states "The 911 and E911 services are not supported." A phone that cannot call 911 is a safety issue. This is a hard disqualifier regardless of its strengths (best multi-PDN, I2S audio, ~1.1 mA idle, carrier certs). Supply risk (Trasna transition) was the initial concern but is secondary — the module is in mass production and available at DigiKey. No built-in GNSS. Small community.
    2. **EC25-AF — strong alternative, not worth switching**: T-Mobile voice certified (unlike EG25-G which is data-only), B71, optional GNSS, 911/eCall supported, multi-PDP with IMS auto-isolated. Available at DigiKey (Mini PCIe ~$60). However: no breakout with integrated codec (slower prototyping), higher idle current (23.3 mA vs 17.5 mA), and the multi-PDP advantage is irrelevant (simultaneous VoLTE+data not required). **Documented as fallback for final PCB if SIM7600 testing reveals a blocking issue.**
    3. **EG25-G T-Mobile VoLTE — clarified**: The EG25-G's VoLTE CS fallback on T-Mobile is a carrier provisioning issue (T-Mobile certifies EG25-G for data only, not voice), not a hardware limitation. The EC25-AF is the proper Quectel alternative (T-Mobile voice certified).
    4. **SIM7600 PDP context conflict — RESOLVED (not a blocker)**: The conflict is confirmed in the `AT+NETOPEN` code path only. This project uses CMUX+PPP (`ATD*99***<cid>#`), which bypasses `AT+NETOPEN` entirely. CMUX+PPP is the standard documented pattern for simultaneous data + AT commands on the SIM7600 (Linux n_gsm, RT-Thread, ESP-IDF). Whether data stays active *during* a live VoLTE call via PPP is unverified but likely (3GPP bearers should coexist). Fallback: pause data during calls — acceptable.
    5. **Simultaneous VoLTE+data — NOT a requirement**: MVP is make/receive calls. Daily driver adds SMS/contacts/menu. None need data during a call. The future ecosystem (car tethering while on a call) is the only scenario, and "pause data during call" is acceptable there too. This was incorrectly treated as a near-term constraint in the first revisit; corrected.
  - *Why SIM7600 wins*: Waveshare NA-H HAT with codec (fast prototyping), built-in GNSS, large community, lower idle current (17.5 mA vs EC25-AF's 23.3 mA), 911 supported. The one advantage EC25-AF had (better multi-PDP) addresses a need the project doesn't have.
  - *Fallback*: EC25-AF for the final PCB if SIM7600 HAT testing reveals a blocking issue. Both are Cat 4, T-Mobile voice certified, B71-capable — switching at PCB time is low-cost.
  - *Tradeoff*: The SIM7600's `AT+NETOPEN` bug is real but irrelevant to this project's CMUX+PPP architecture. Data-during-VoLTE-call is unverified via PPP; if it fails, pause data during calls. No modem switch needed.

### 2026-06-28: MCU Selection
- **Decision**: Select STM32H743 (Cortex-M7 @ 480MHz) as the MCU, in LQFP-144 package (STM32H743ZI).
  - *Rationale*: Four MCU candidates were evaluated against the "no 5+ features blocked" principle and the concurrent processing load requirement:
    - **nRF52840**: Eliminated — no camera interface (blocks photo capture 6, video 5), I2S clock accuracy problems, USB device only (no OTG), 64MHz is tight for MP3 decode.
    - **ESP32-S3**: Viable but with Zephyr USB support only 8 months old (risk for ecosystem tethering), and BLE advantage is moot since neither ESP32-S3 nor STM32 has classic BT (A2DP) — external BT module needed either way.
    - **ESP32-P4**: Most capable for multimedia (hardware H.264, MIPI camera/display) but too new — limited availability, immature Zephyr support, no built-in wireless, likely BGA-only. Too risky for first custom PCB.
    - **STM32L4R5**: Questionable on processing power — 120MHz cacheless M4 running Zephyr's full networking + USB + LVGL + audio stack concurrently is tight, especially for ecosystem tethering (PPP routing + USB CDC ECM + audio + display).
    - **STM32H743**: 480MHz Cortex-M7 with L1 cache (16KB I + 16KB D), hardware JPEG codec, USB OTG FS + HS, DCMI camera, 4x SAI audio, 2x SDMMC, LTDC display controller. Handles the worst-case concurrent load (tethering + music + UI + call) comfortably. ARM Cortex-M7 (career-relevant). Mature Zephyr support (multiple H7 boards in mainline). ~$12-15, within budget.
  - *Processing power analysis*: The real CPU demand isn't any single feature — it's the concurrent subsystem load. Worst case: ~~ecosystem tethering (USB CDC ECM + PPP routing + potentially NAT)~~ [superseded 2026-06-28 — tethering now runs on the modem's USB, not the MCU; MCU load for tethering is ~zero], MP3 decode, LVGL rendering, and modem AT command management all running simultaneously. This is essentially a media player + UI + phone (the router role moved off the MCU). The H7's 480MHz + L1 cache is still the right class of part for this; cacheless M4 parts (L4) would struggle with the remaining media + UI + modem load.
  - *Package choice (LQFP-144)*: The H743 comes in LQFP-100, -144, and -176 (all 0.5mm pitch). LQFP-100 loses USB HS (ULPI requires 12 dedicated pins that don't fit). LQFP-144 retains USB HS and all needed peripherals. LQFP-176 adds little for this project. LQFP-144 is hand-solderable with practice; JLCPCB assembly is the fallback (and will be needed for the SIM7600 LGA module regardless).
  - *Tradeoffs*: Higher power consumption (~20 µA sleep vs 0.4 µA for L4) — but modem dominates at 17,500 µA (17.5 mA LTE idle/DRX, verified 2026-06-28), so MCU sleep current is negligible. Larger package than L4. No built-in BLE — external BT module needed (same as any STM32 path, and same as ESP32-S3 for classic BT).
  - *USB HS note*: ~~True USB High-Speed (480 Mbps) requires an external ULPI transceiver (e.g., USB3300, ~$2). Can start with USB FS (built-in PHY) for MVP and populate ULPI transceiver later for ecosystem tethering. LQFP-144 keeps this option open.~~ **SUPERSEDED 2026-06-28**: USB3300 ULPI dropped — the SIM7600's own USB 2.0 HS port does tethering directly, bypassing the MCU. MCU USB OTG_FS is sufficient. LQFP-144 retained for GPIO margin. See "USB HS / ULPI Revisit" decision below.
- **Decision**: Bluetooth will be added as an external module (not built into MCU).
  - *Rationale*: Neither STM32H7 nor ESP32-S3 has classic Bluetooth (A2DP for audio streaming). Both have BLE (ESP32-S3 built-in, STM32 via external module). Since an external BT module is needed for classic BT regardless of MCU choice, the STM32's lack of built-in BLE is not a disadvantage. ### 2026-06-28: Codec Selection / Audio Path Architecture
- **Decision**: Select **MAX9880A** (Maxim/ADI, TQFN-48, ~$1.70) as the audio codec. Dual-port architecture: SIM7600 PCM → MAX9880A primary port (voice), STM32H743 I2S → MAX9880A secondary port (music). **MCU is NOT in the voice audio path during calls.** LOCKED 2026-06-28.
  - *Rationale*: The revisit was triggered by a HIGH PRIORITY open question: the docs claimed a single codec (e.g., WM8960) "unifies" voice (PCM from module) and music (I2S from MCU) into one audio path, but most codecs appeared to be I2S-only with no second PCM port. Three research threads confirmed:
    1. **SIM7600 is PCM-only** (Hardware Design Manual V1.03 §3.6): fixed master mode, short-frame sync, 16-bit linear, 2048/4096kHz clock. No I2S mode, not configurable via AT commands. The Waveshare HAT's NAU8810 is wired in PCM mode to match.
    2. **All common codecs have ONE digital audio port** — WM8960, NAU8810, NAU8822, SGTL5000, TLV320AIC3204/3104, ES8316/8388. They can be configured for I2S *or* PCM, but not both simultaneously. The original "single codec unifies voice and music" claim was **false** for all these parts. The TLV320AIC3106's "alternate bus" is pin multiplexing, not a second independent port (confirmed by TI forum).
    3. **MAX9880A is the only hobbyist-accessible dual-port codec** — two truly independent digital audio interfaces (primary for voiceband PCM/TDM, secondary for stereo I2S/TDM), running simultaneously and asynchronously. ~$1.70 in TQFN-48 (hand-solderable with hot air). Designed specifically for smartphones (voice + media). The TLV320AIC34 is also dual-port but BGA-only (not hobbyist-friendly).
  - *Why MAX9880A over MCU bridge*: The alternative was an MCU bridge (SAI1 PCM slave from SIM7600 → SAI2 I2S master to single-port codec). This is feasible on the H743 (~8-10ms latency, <1% CPU, well within ITU G.114) but has two problems: (a) the MCU is always in the voice audio path, adding latency and firmware complexity (real-time DMA bridging, cache coherency, codec rate switching between 8kHz calls and 48kHz music); (b) Zephyr's STM32H7 SAI driver is immature (PR #92887 in progress, not confirmed merged) — could require driver porting or direct HAL usage. The MAX9880A avoids both issues: voice goes directly module→codec (MCU not involved), and the MCU I2S for music is simpler and more tolerant. The MAX9880A is also cheaper ($1.70 vs WM8960 $6-12 obsolete) and matches real smartphone architecture.
  - *Why not two codecs* (NAU8810 for voice PCM + PCM5102 for music I2S): This is the "two audio paths" problem originally rejected — needs an analog mixer/switch to combine outputs to the speaker, more components, more PCB complexity. The MAX9880A does the mixing internally. Eliminated.
  - *Pre-PCB verification items*: (1) Confirm MAX9880A primary port supports PCM short-frame sync, 16-bit, master clock from module (check datasheet — subagent reported "TDM and I2S formats"; PCM short-frame is a common TDM subset but must be confirmed). (2) Verify stock availability at DigiKey/Mouser (Maxim/ADI parts can have long lead times). (3) MAX9880A is 1.8V supply — add 1.8V LDO and level shifter for MCU's 3.3V I2S lines (unidirectional, simple). (4) MAX9880A is NOT on the Waveshare HAT (HAT has NAU8810) — committed for final PCB, not prototyped on HAT.
  - *Fallback*: If MAX9880A is unavailable or PCM-incompatible, fall back to MCU bridge architecture with NAU8822 (single-port I2S codec, ~$1.20). SAI1 as PCM slave, SAI2 as I2S master. Run codec at 8/16kHz during calls, 48kHz for music. Requires Zephyr H7 SAI driver. See research-notes.md Codec Selection Finding 3.
  - *Prototyping impact*: The Waveshare NA-H HAT prototyping step is **unchanged** — it validates the SIM7600's PCM voice output on T-Mobile VoLTE using the NAU8810. This is the riskiest unknown and is independent of the final codec choice.
  - *Tradeoff*: The MAX9880A is less documented in the hobbyist community than WM8960/NAU8810. The 1.8V supply adds a small amount of power-supply complexity (one LDO + one level shifter). These are minor compared to the firmware complexity of the MCU bridge alternative.

### 2026-06-28: Display Selection
- **Decision**: Select **ST7789V SPI color TFT** (2.0" 240×320, 4-wire SPI, RGB565, ~$5–8) as the display. LOCKED 2026-06-28.
  - *Rationale*: The display decision was triggered by the approaching schematic phase (display interface affects PCB routing and GPIO allocation) and the "no 5+ features blocked" principle — the display must be color-capable (photo capture rated 6, camera preview rated 6, video recording rated 5; a monochrome display would block these). Five options were evaluated:
    1. **SPI color TFT (ST7789V, 2.0" 240×320)** — SELECTED. 6 GPIO pins, 150KB framebuffer fits internal 1MB SRAM (no external SDRAM), ~$5–8, ~30–50mA with backlight (off during standby — modem dominates at 17.5mA). Zephyr `display_st7789v.c` is the most mature SPI display driver in main tree; LVGL has native ST7789 support. Camera preview ~15–25fps partial updates (adequate for rated-6 casual photos). 240×320 @ 2.0" ≈ 200 PPI — standard feature-phone resolution.
    2. **SPI color OLED (SSD1351, 1.5" 128×128)** — rejected. **Actually higher power than TFT** (~57–71mA: VCI 23–29mA + Vcc 33–42mA per datasheet — self-emissive OLED draws current per pixel; for bright text/menus it's less efficient than a backlit TFT). 3–5x more expensive ($15–24 vs $5–8). 128×128 too low for phone UI/camera preview. Not in Zephyr main tree (issue #88923 open; only out-of-tree driver nktsb/zephyr-ssd1351-driver). SSD1327 is in Zephyr but grayscale, not color. No larger color OLED options exist at reasonable prices.
    3. **E-ink (B/W or spot-color BWRY)** — **DISQUALIFIED** by "no 5+ blocked" on two grounds: (a) refresh rate makes camera preview impossible — monochrome partial refresh ~0.3s (3fps best case), color full refresh 12–22 seconds; camera preview (rated 6) needs ~15+ fps; (b) "color" e-ink at 1.5–2.4" is spot-color only (black/white/red/yellow — no blue, no green, no gradients), cannot display photos; true full-color e-ink (ACeP) starts at 6"+ and $100+. E-ink's advantages (ultra-low power, bistable, sunlight readable) are genuine but don't outweigh blocking three 5+ features.     4. **Parallel RGB TFT via LTDC (2.0" 320×480)** — rejected. 20 (RGB565) or 28 (RGB888) GPIO pins — eats the spare-GPIO margin on LQFP-144 needed for future peripherals (BT 4 pins, modem USB 2 pins; ~~ULPI 12 pins~~ dropped 2026-06-28). ### 2026-06-28: USB HS / ULPI Revisit — Architecture Change to Modem-Direct USB Tethering

- **Decision**: **Drop the USB3300 ULPI transceiver / MCU USB OTG_HS path.** Use the SIM7600's own USB 2.0 HS port for ecosystem tethering (RNDIS/ECM via `AT+CUSBPIDSWITCH`), bypassing the MCU entirely. Use the MCU's USB OTG_FS (12 Mbps, built-in PHY) for all MCU-originated USB (firmware updates, file/MSC transfer, CDC ACM debug).
  - *Rationale*: Two findings drive this:
    1. **Zephyr STM32H7 USB OTG_HS + ULPI is viable but no longer needed.** ULPI support is in mainline Zephyr (PR #52772, merged Dec 2022; issue #52420 closed/completed 2024-05-14; bugs #61464 ECM-assertion and #75179 new-stack-ULPI both fixed by #104825/#76115). The STM32H747I-DISCO is a mainline board using ULPI (USB3300), confirmed working with the `cdc_acm` sample. The scary "H743 HS can't receive data" bug (#57499) was **hardware** — a floating USB3300 IO power rail; the reporter fixed it and confirmed "USB reliable and matches dev boards." So ULPI works, but it is the less-trodden path (only one mainline H7 board uses it; clock config is confusing — issue #71248 has open TODOs; ULPI pinctrl slew rate must be "high-speed"; new UDC stack is less battle-tested for ULPI than FS).
    2. **The SIM7600's own USB port does tethering directly and better.** The SIM7600 has a USB 2.0 HS (480 Mbps) device port (SIMCom hardware design PDF: "USB2.0 is up to 480Mbps"). It presents itself as a USB network adapter via RNDIS (`AT+CUSBPIDSWITCH=9011,1,1`) or ECM (`=9018,1,1`), simultaneously exposing AT/ttyUSB serial ports (composite device — confirmed via `lsusb` on Raspberry Pi). This is the module's native, best-supported tethering path (used by RPi, routers), independent of the `AT+NETOPEN` bug and independent of the project's CMUX+PPP-over-UART path. The modem's USB is free to be dedicated to tethering because the MCU controls the modem via UART.
  - *Architecture impact*: For ecosystem tethering, the car module (USB host) connects to the modem's USB (USB device) and gets LTE at 480 Mbps — no MCU bridging, no ULPI, no USB3300, no Zephyr USB HS stack, no 12 GPIO pins. The MCU USB FS bottleneck only existed when the MCU bridged LTE data to CDC ECM; that bridge is eliminated. MCU USB FS (12 Mbps) is sufficient for firmware, file transfer, and debug.
  - *Topology*: The phone has two USB device ports (MCU OTG_FS, modem USB HS) and the car module is a host. Connecting two devices to one host over one cable requires a USB hub. **Rev1 (minimal commitment)**: route the modem USB D+/D- to a connector footprint or test points; MCU USB on the main USB-C. **Future ecosystem respin**: add an internal USB 2.0 hub (e.g., USB2514, ~$1–2) so a single USB-C presents both the modem (RNDIS for LTE) and the MCU (MSC for files) to the car module — simultaneous LTE + file access over one cable.
  - *Supersedes*: The "USB HS via external ULPI transceiver (USB3300) is a board-level upgrade path preserved by the LQFP-144 package" claim (requirements.md Resolved Questions "USB mode", research-notes.md USB HS/ULPI Clarification practical path). LQFP-144 is retained — it was also chosen for its GPIO margin (now 41 spare after dropping ULPI), camera bit depth, and overall peripheral fit, not ULPI alone.
  - *Tradeoff*: Defers the hub topology to a future respin (tethering is rated 6, post-daily-driver scope). The modem USB routing footprint on rev1 keeps the door open at near-zero cost. If a future use case ever needs the MCU itself to do HS USB (not tethering), the ULPI path remains available on LQFP-144 and Zephyr supports it — but that is not a current requirement.
  - *Pre-PCB / prototype verification*: (1) Validate `AT+CUSBPIDSWITCH=9018,1,1` (ECM) and `=9011,1,1` (RNDIS) on the Waveshare NA-H HAT connected to a Linux host — confirm enumeration as a network adapter and LTE data flow (cheap HAT test). (2) Confirm whether the modem can run PPP-over-UART (MCU data) simultaneously with RNDIS-over-USB (car module data) — two PDP contexts; likely yes but unverified. See research-notes.md "USB HS / ULPI Revisit" section.

### 2026-07-18: Keypad Integration — gpio-kbd-matrix + Dial String UI

- **Context**: Post-MVP roadmap (2026-07-13) identified 4×4 matrix keypad + arbitrary-number dialing as the next step. User has the Adafruit 4×4 membrane keypad (PID 3844) on hand. This session implements keypad input, dial string accumulation, and replaces the hardcoded `CALL_NUMBER` with a user-typed number.
- **Decision**: Use Zephyr's built-in `gpio-kbd-matrix` input driver (not a manual matrix scan).
  - *Rationale*: The driver handles scanning, per-key debouncing, ghost-key detection, and idle wake (interrupt mode) in its own thread. The app just receives `(col, row)` events via an input callback. This is the idiomatic Zephyr approach (the old `kscan` API was renamed to `input` in recent Zephyr), teaches the input subsystem (aligned with the project's learning goal), and requires **no firmware rework when moving to the custom PCB** — only the devicetree GPIO pins change. A manual scan would reinvent debounce/ghost-detection and be thrown away at PCB time.
- **Decision**: Pin assignment for the 4×4 matrix on the Nucleo Arduino headers (user-selected):
  - Rows: R1→D3 (PE13), R2→D4 (PE14), R3→D5 (PE11), R4→D6 (PE9) — inputs, pull-up, active-low
  - Cols: C1→A1 (PC0), C2→A2 (PC3), C3→A3 (PB1), C4→D2 (PG14) — outputs, active-low
  - *Rationale*: All 8 pins are free (not used by SPI1 display, LPUART1 modem, I2C1, USART3 console, B1 button, or ADC1 on A0). A1–A3 are Arduino analog-header pins but used as digital GPIO (the keypad needs digital I/O, not ADC). All 4 row pins support EXTI interrupts (required for the driver's `idle-mode = "interrupt"` wake).
- **Devicetree**: Added `kbd_matrix` node to `app.overlay` with `compatible = "gpio-kbd-matrix"`, 4 `row-gpios` + 4 `col-gpios`, `debounce-down-ms = <10>`, `debounce-up-ms = <20>`, `poll-timeout-ms = <100>`, `idle-mode = "interrupt"`. Added `CONFIG_INPUT=y` to `prj.conf` (the driver auto-selects via `DT_HAS_GPIO_KBD_MATRIX_ENABLED`, but the explicit config documents intent). Build verified: the driver bound to the node (`gpio_kbd_matrix_cb_0` + `gpio_kbd_matrix_init` in the linker map), devicetree compiled with correct pin flags (`0x11` = `GPIO_PULL_UP | GPIO_ACTIVE_LOW` on rows, `0x1` = `GPIO_ACTIVE_LOW` on cols).
- **Firmware architecture** (main.c):
  - **Input callback** (`keypad_callback`): registered via `INPUT_CALLBACK_DEFINE(DEVICE_DT_GET(DT_NODELABEL(kbd_matrix)), ...)`. Runs in the input subsystem thread (not ISR, not main loop). Captures `INPUT_ABS_X` (col) and `INPUT_ABS_Y` (row) on the ABS events, then on `INPUT_BTN_TOUCH` press (value=1, sync=true) looks up `keymap[row][col]` and puts the character into `key_msgq` via `k_msgq_put(..., K_NO_WAIT)`.
  - **Key event queue** (`K_MSGQ_DEFINE(key_msgq, sizeof(char), 16, 4)`): passes key characters from the input callback (driver thread) to the main loop (owns LVGL + modem). This is the thread-safety boundary — the callback never touches LVGL or the modem UART directly, avoiding cross-thread LVGL/modem calls. `k_msgq` is designed for cross-thread communication.
  - **Keymap** (`keymap[4][4]`): maps `(row, col)` → character per the Adafruit 4×4 membrane layout: row 0 = `1 2 3 A`, row 1 = `4 5 6 B`, row 2 = `7 8 9 C`, row 3 = `* 0 # D`.
  - **Dial string** (`dial_string[21]` + `dial_len`): replaces the hardcoded `CALL_NUMBER`. Digits `0-9/*/#` append (IDLE state only, max 20 chars), `C` backspaces, `A` triggers `ATD<dial_string>;` via `snprintf`, `B` hangs up / cancels / rejects, `D` is a no-op (menu, future).
  - **handle_key(char key)**: replaces `handle_button()`. State-dependent key actions: IDLE → digits append / A=call / C=backspace; CALLING → B=cancel; RINGING → A=answer / B=reject; CONNECTED → B=hang up.
  - **B1 button removed**: the Nucleo's user button (PC13) was the previous call control. Now redundant — A/B keys do the same things. Removed `gpio_dt_spec`, `gpio_callback`, `button_isr`, button init, and button polling from the main loop. Also removed `#include <zephyr/drivers/gpio.h>` (no longer needed — the input driver handles GPIO internally).
  - **Main loop**: drains `key_msgq` first (`k_msgq_get` with `K_NO_WAIT`), then processes modem URCs (`modem_readline`), then `lv_timer_handler()`. All LVGL and modem access stays in the main loop — no cross-thread issues.
- **Thread safety**: The input callback runs in the input subsystem thread. It only does `k_msgq_put()` (thread-safe by design). All LVGL calls, modem UART calls, and `call_state` transitions happen in the main loop (which drains the queue and processes URCs). `call_state` is read/written only in the main loop now — no race. The only cross-thread data path is `key_msgq`, which is purpose-built for that.
- **Build result**: Firmware compiles clean (exit 0, no warnings). Flash 333,956 B (15.92%), RAM 56,980 B (10.87%). Delta from before: +292 B flash (keymap, callback, handle_key, msgq plumbing), +80 B RAM (msgq buffer + dial string). The input driver + matrix library add ~5 KB flash (already counted in the first build with the overlay).
- **Not yet tested on hardware**: The firmware compiles and the driver binds, but the keypad wiring + functional test (press key → digit appears on display → type number → press A → call places) is the next step. The user has the keypad on hand.
- **Hardware test — FIXED and VERIFIED same day**: First flash showed `<err> input_gpio_kbd_matrix: Pin 0 interrupt configuration failed: -16` (EBUSY). Root cause: STM32 EXTI line conflict — our row 0 is PE13 (needs EXTI13), but the board DTS defines the B1 user button on PC13 (also EXTI13). STM32 EXTI lines 0–15 are port-shared (only one port per line), and the `gpio-keys` driver claimed PC13/EXTI13 first at boot, blocking PE13. Fix: added `&user_button { status = "disabled"; }` to `app.overlay` — we removed B1 from app code (keypad A/B replaced it), so disabling the board's button node frees EXTI13 for PE13. Build verified the `status = "disabled"` propagated to the compiled DTS. After reflash: clean boot, no `<err>` lines, keypad fully functional.
- **Functional test confirmed**: User typed `6077936142` (with a `123` typo cleared via C/backspace), pressed A → `ATD6077936142;` → `VOICE CALL: BEGIN` (connected in ~10s) → pressed B → `AT+CHUP` → `VOICE CALL: END: 000044`. **Real VoLTE call to an arbitrary user-typed number, no PC involvement, controlled entirely from the keypad + display.** This is the standalone-dialer milestone — the phone is no longer a hardcoded test jig.

### 2026-07-19: Codec Swap MAX9880A → Realtek ALC5651 (LOCKED)

- **Why the ALC5651 is a drop-in replacement**:
  - **Dual I2S/PCM interface** — same architecture as MAX9880A: I2S-1 for PCM voice from SIM7600 (primary), I2S-2 for I2S music from MCU (secondary). Both simultaneous and asynchronous (ASRC per interface).

ALC5651 §7.5.1 supports PCM Mode A (short sync, Figures 10-11) in slave mode — exact match. The ALC5651 can be PCM slave on I2S-1 (SIM7600 is master) and I2S slave on I2S-2 (MCU is master).
  - **Better audio specs**: 100dBA DAC SNR (vs MAX9880A 96dB), 94dBA ADC SNR (vs 82dB).
  - **Smaller package**: QFN-40 5×5mm (vs MAX9880A TQFN-48 6×6mm).

### 2026-07-19: L1 Inductor Selection — Coilcraft XFL4020-152MEC (LOCKED)

- **Context**: Revisit prompt item B7 (L1 power inductor for TPS63021DSJR buck-boost). This is the last remaining component in the 3.3V power rail — the most critical net on the board. The inductor has hard electrical constraints from the TPS63021DSJR datasheet (TI SLVS916I): inductance, saturation current, DCR. The revisit prompt noted the datasheet-recommended part was Coilcraft XFL4020-152ML (4×4mm) and asked to verify specs against the datasheet and check JLC availability.
- **Datasheet sourced**: The TPS63021DSJR datasheet was NOT in `docs/reference/`. Downloaded TI SLVS916I Rev. I (Oct 2019) from https://www.ti.com/lit/ds/symlink/tps63021.pdf → `docs/reference/tps63021.pdf` (1.86 MB). Added to `docs/reference/README.md` index.
- **Specs verified from datasheet** (via PDF MCP server, §8.2.2.2 Inductor Selection + Table 2 + Table 4):
  - *Inductance*: 1.5µH recommended for 1.5–2A output (1µH also acceptable; 2.2µH only for lower loads). Internal loop compensation requires L ≥ 1µH to avoid subharmonic oscillation.
  - *Saturation current*: datasheet §8.2.2.2 says "choose an inductor with saturation current 20% higher than the calculated peak" (Eq 2, boost mode is the critical case). The two recommended parts in Table 2 have Isat of 5.1A (Coilcraft XFL4020-152ME) and 5.4A (muRata FDV0530S-H-1R5M) — both well above the 4A switch current limit.
  - *DCR*: as low as possible (efficiency). Table 2: 15mΩ (Coilcraft), 24mΩ (muRata).
  - *Switching frequency*: 2.5 MHz typical (fixed internal).
  - *Table 4* (components for VOUT=3.3V characterization curves): L1 = 1.5µH 4×4×2mm Coilcraft XFL4020-152ML (the "L" = low-DCR variant).
- **Decision**: **L1 = Coilcraft XFL4020-152MEC (LCSC C3033018). LOCKED 2026-07-19.**
  - *Specs*: 1.5µH ±20%, 4×4×2.1mm, magnetic shielded. Isat 4.4A at 20% inductance drop (Coilcraft datasheet authoritative). DCR 14.4mΩ typ / 15.8mΩ max. Irms 9.1A (40°C rise). 757 in stock at LCSC, JLC "Extended" (JLC sources from LCSC on demand — same status as our display panel).
  - *Why the MEC variant* (not the ML from Table 4): The "ML" low-DCR variant is NOT on LCSC (broker-only, e.g., ic2world.com). The "MEC" standard-DCR variant IS on LCSC (C3033018). DCR 14.4mΩ is essentially the same as the datasheet's 15mΩ spec for the ME variant — the "L" low-DCR variant would be marginally lower but is unobtainable for JLC assembly.
  - *Why lowest DCR matters*: The 3.3V rail powers everything (MCU, display, SD, codec I2S side, level shifter). DCR conduction losses are I²R — at 2A average, 14.4mΩ = 58mW lost; the Taiyo Yuden alternative at 35mΩ = 140mW lost (2.4× more). On a battery-powered phone, this directly hurts standby and talk time.
  - *Why magnetic shielded*: The inductor sits near the modem/audio/RF section. Unshielded inductors radiate switching noise (2.5 MHz + harmonics) that can couple into audio and RF. Shielded construction contains the field.
- **Isat adequacy analysis** (the one judgment call):
  - The TPS63021's switch current limit is 4A typical — a fault condition, not the operating current. The inductor physically cannot see more than 4A through it (the IC enforces this).
  - The datasheet's "20% above calculated peak" guidance → wants Isat ≥ 4.8A. The XFL4020-152MEC's 4.4A is below this strict target.
  - **Why 4.4A is adequate for our use case**:
    - *Buck mode* (Vin=4.2V, Vout=3.3V, full battery): peak inductor current at 3A out ≈ Iout + ΔIL/2 = 3 + 0.095 = 3.1A. Well under 4.4A.
    - *Boost mode* (Vin=3.0V, Vout=3.3V, near-dead battery): using Eq 2 at 3A out, Iavg_switch = Iout·Vout/(Vin·η) = 3·3.3/(3.0·0.9) = 3.67A, Ipeak ≈ 3.7A. Under 4.4A but only 19% margin — close to the 20% recommendation.
    - *Realistic phone loads*: peak 2–3A (modem TX burst + display + audio + MCU), not sustained 3A. The modem alone is the big draw and it's duty-cycled. Even in boost mode at 3A, we're under 4.4A. The phone won't sustain 3A in boost mode (battery at 3.0V is nearly dead — the modem will be duty-cycling and the user will be charging soon).
  - *Conclusion*: 4.4A Isat is adequate. The 4A switch limit is the hard ceiling; our operating currents are well below it. If we ever needed sustained 3A in boost mode, we'd want the higher-Isat alternative — but that's not a phone use case.
- **Alternative rejected**: Taiyo Yuden LSDND4040WKT1R5MM (C6653414) — Isat 7A (huge margin) but DCR 35mΩ (2.4× higher → noticeable efficiency loss on the most critical rail). The Isat margin isn't worth the efficiency hit for a battery-powered phone. Would revisit only if sustained 3A+ boost-mode operation became a requirement (it isn't).

  - DMBJ PNLS4012-1R5M (C2849458): Isat 2.1A — way too low.

- **Decision 2 — TPS63021 VINA wiring CORRECTED**: ~~VINA wired to VBAT with 0.1µF bypass cap~~ **CORRECTED 2026-07-21**: VINA's only external connection is a 0.1µF cap to GND — **no wire to VBAT/VIN**. - **Decision 3 — TPS63021 EN pin: 1MΩ pullup to VBAT**: EN is the chip's on/off switch (high=run, low=shutdown ~1µA). **Cannot tie to VOUT** — chicken-and-egg at startup (VOUT=0 → EN=0 → chip never starts). **Cannot leave floating** — datasheet says "must not be left open" (§5). **Decision: 1MΩ pullup to VBAT** (always-on whenever battery is connected). Direct wire to VBAT was considered and rejected — locks out future MCU control for ~$0.001 savings.
- **Decision 4 — TPS63021 PS/SYNC: direct wire to GND**: PS/SYNC controls power-save mode (low=enabled, high=disabled). **Decision: tie directly to GND** (no resistor, no cap). Power-save enabled → at light load (<100mA) chip enters burst mode (high efficiency, ~25µA Iq); at heavy load (>100mA) chip auto-switches to PWM (low ripple). Transition is automatic — no MCU toggling needed. Standby load (~20-50mA: modem idle 17.5mA + MCU sleep + fuel gauge) benefits from power-save; active load (>100mA: display + modem TX) auto-switches to PWM. No cap on digital control pins (caps are for debounce/edge-rate control; PS/SYNC is wired to a static logic level).
- **Decision 5 — TPS63021 PG pin: still OPEN**: Power-good output (open-drain, low=fault). Two options: (a) leave open (chip works fine without it); (b) 10kΩ pullup to +3V3 + wire to MCU GPIO for rail-failure detection. **Not yet decided — open question for next session.**

### 2026-07-21: Remaining Power Open Questions Resolved (PG, EN, ALRT, I2C, R_PROG)

- **Context**: User resolved all 5 remaining open questions from the 2026-07-21 power pinout session in one batch. All decisions LOCKED.
- **Decision 1 — TPS63021 PG pin: wire to MCU (LOCKED)**: User's initial intuition was "if power not good, MCU is off → no point wiring." Corrected: the TPS63021 PG threshold is ~90% of VOUT (~2.97V), but the STM32H743 BOR triggers around ~1.7V — there's a ~1.2V window where PG is low but the MCU is still alive and can take defensive action (reduce clock, kill backlight, pause modem TX, graceful shutdown). Without PG, the MCU has no warning before the rail collapses. Wiring: 10kΩ pullup to +3V3, net `PG_3V3`, MCU GPIO (EXTI-capable, TBD pin). Cost: 1 GPIO + 1 resistor.
- **Decision 2 — TPS7A02 EN pin: tie to IN (LOCKED)**: +1V8 comes up whenever +3V3 is up. No MCU GPIO control. Rationale (from ALC5651 datasheet §7.1): "to prevent all power down leakage, needs keep all power supply on" — cutting 1.8V while 3.3V rails stay up *causes* leakage. Standby power is handled via I2C register power management (ALC5651 §7.16, MX-61h through MX-66h), not by cutting rails. The recommended power-on sequence (1.8V before 3.3V) is architecturally impossible with our 1.8V-from-3.3V topology; the codec's POR handles the ramp. See project-log.md 2026-07-21 TPS7A02 EN discussion for full reasoning.
- **Decision 3 — MAX17048 ALRT pin: wire to MCU (LOCKED)**: Open-drain active-low alert → MCU GPIO (EXTI-capable, TBD pin) + 10kΩ pullup to +3V3. Net `ALRT_BATT`. Fires on low-SOC, 1% SOC change, battery under/overvoltage, VRESET alert. Rationale: low-SOC → UI warning; undervoltage → graceful shutdown before LiPo damage (safety); overvoltage → charger runaway detection (defense-in-depth); interrupt-driven saves power vs polling. Trivial cost (1 GPIO + 1 resistor, 41 spare GPIO available). See project-log.md 2026-07-21 ALRT discussion for full reasoning.
- **Decision 4 — I2C address conflict: NO CONFLICT (RESOLVED)**: Verified both addresses from datasheets. MAX17048 = 0x36 (7-bit, from `docs/reference/max17048.pdf`). ALC5651 = 0x1A (7-bit, from `docs/reference/alc5651.pdf` §7.14.1 Table 14 — address byte `0011010 R/W`, 8-bit write = 0x34). 0x36 ≠ 0x1A — both share the same I2C bus with no collision. Neither device has configurable address pins, so if they had collided it would have required an I2C multiplexer or separate bus — but they don't.
- **Decision 5 — MCP73831 R_PROG = 2kΩ / 500mA charge (LOCKED)**: Target battery is 1200mAh LiPo (per project-log line 165, 179). 500mA = 0.42C, well within the 0.5C standard LiPo charge rate. USB 2.0 standard port provides 500mA — exact match. ~2.5-3 hour charge time. If a larger battery is used later (2000mAh+), 500mA becomes more conservative (0.25C). If a smaller battery is used (<500mAh), R_PROG must be increased to keep charge rate ≤1C. The MCP73831 is capped at 500mA — fast charging (2A+) would need a different IC (BQ25895 etc.), but 500mA is fine for prototype.
